# 手動テストケース — ウィンドウフォーカス復帰

通知クリック時に正しいターミナルにフォーカスが戻るかを検証する。

## 事前準備

```bash
swift build -c release
```

テスト用のショートハンド（各ターミナルで実行）:

```bash
P="pyokotify /tmp/pyokotify-test.png -d 10 --auto-click --cwd $(pwd)"
```

`--auto-click` は通知表示後1秒で自動クリックをシミュレートする。
フォーカスが元のターミナルに戻れば成功。

---

## A. 直接起動

各ターミナルで `$P -t A` を実行し、そのターミナルに戻ることを確認する。

テスト対象: 普段使うターミナル、Ghostty、Terminal.app、VSCode統合ターミナルなど。

---

## B. tmux経由

1. ターミナルで `tmux` を起動
2. tmux内で `$P -t B` を実行
3. 元のターミナルアプリにフォーカスが戻ることを確認

### ペイン復元

1. tmuxで2分割する
2. ペイン0で `$P -t B-pane` を実行
3. すぐにペイン1に切り替える
4. 自動クリック後、ペイン0が選択されることを確認

### ウィンドウ復元

1. tmuxで2ウィンドウ作成
2. window 0で `$P -t B-win` を実行
3. すぐにwindow 1に切り替える
4. 自動クリック後、window 0に切り替わることを確認

---

## C. zellij経由

1. ターミナルで `zellij` を起動
2. zellij内で `$P -t C` を実行
3. 元のターミナルアプリにフォーカスが戻ることを確認

zellijは `TERM_PROGRAM` を上書きしないため、親ターミナルの値で検出される。

---

## D. クロスターミナル

同じプロジェクトディレクトリを2つのターミナルで開いた状態でテストする。
ウィンドウタイトルではなくプロセスツリーで正しいターミナルを区別できるかの検証。

1. ターミナルAとターミナルBで同じディレクトリを開く
2. ターミナルAで `$P -t D-A` を実行 → ターミナルAに戻ることを確認（Bではなく）
3. ターミナルBで `$P -t D-B` を実行 → ターミナルBに戻ることを確認（Aではなく）

tmux/zellij経由でも同様に確認する。

---

## E. VSCode固有

### 複数ウィンドウ

1. VSCodeでプロジェクトAとプロジェクトBを別ウィンドウで開く
2. プロジェクトAのターミナルで `$P -t E-multi` を実行
3. プロジェクトAのウィンドウに戻ることを確認

### worktree

1. git worktree内でVSCodeを開く
2. `$P -t E-wt` を実行
3. 親リポジトリのVSCodeウィンドウに戻ることを確認

---

## F. エッジケース

- 短いディレクトリ名 (`/tmp/a`) で実行
- 日本語ディレクトリ名 (`/tmp/テスト`) で実行
- スペース入りパス (`/tmp/my project`) で実行
- 存在しないcwd (`--cwd /nonexistent/path`) でクラッシュしないこと
- cwdなし（`--cwd` 省略）でクラッシュしないこと
- `log stream --predicate 'subsystem == "com.haryoiro.pyokotify"' --level debug` でウィンドウ検出ログが出力されること

---

## G. Claude Code Hooks連携

Claude Codeのhooks経路で通知が正しく動作するかを検証する。

1. 対象ターミナルでClaude Codeを起動
2. 会話を終了して Stop イベントを発生させる
3. 通知にプロジェクト名・ブランチが表示されることを確認
4. 通知クリック（または `--auto-click`）で元のターミナルに戻ることを確認

tmux/zellij内のClaude Codeでも同様に確認する。
